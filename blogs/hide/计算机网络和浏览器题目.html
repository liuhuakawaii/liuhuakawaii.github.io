1. HTTP 请求方式 ​

HTTP 请求方法主要包括：

  • GET: 获取资源，通过URL传递参数，是安全且幂等的方法，响应可被缓存（受Cache-Control控制）
  • POST: 提交数据创建新资源，数据在请求体中，非幂等方法，响应通常不被缓存
  • PUT: 完整替换或创建指定资源，是幂等方法，请求体包含完整的资源数据
  • DELETE: 删除指定资源，是幂等方法，通过URL指定要删除的资源
  • HEAD: 获取资源的元信息（首部），与GET类似但不返回响应体，是安全且幂等的方法
  • OPTIONS: 获取目标资源支持的通信选项，常用于CORS预检请求
  • PATCH: 对资源进行部分修改，非幂等方法，请求体包含部分更新的数据
  • TRACE: 追踪请求-响应的传输路径，主要用于诊断
  • CONNECT: 建立网络隧道，主要用于HTTPS等加密通信的代理请求

2. Get/Post 的区别 ​

GET和POST的主要区别：

  1. 语义区别：
  • GET用于获取资源，不应该对服务器状态产生影响
  • POST用于提交数据，可能会修改服务器状态
  1. 幂等性：
  • GET是幂等的，多次请求结果相同
  • POST非幂等，每次请求可能产生不同结果
  1. 缓存：
  • GET请求默认可被浏览器缓存
  • POST请求默认不被缓存，但可通过正确配置启用缓存
  1. 数据传输：
  • GET参数通过URL传递，会显示在浏览器地址栏
  • POST数据在请求体中传输，对用户不可见
  1. 数据限制：
  • GET受URL长度限制（具体限制因浏览器和服务器而异）
  • POST理论上无限制，实际受服务器配置限制
  1. 使用场景：
  • GET适用于数据查询、资源获取
  • POST适用于数据提交、状态修改

注意：两种方法在HTTP层面的安全性是一样的，都需要通过HTTPS来保证传输安全。

其实从接口定义的角度来说，GET和POST没有规定参数放在URL还是Body中。只是为了规范来说我们约定俗成的把参数位置区分开来。

3. RESTful 规范 ​

REST (Representational State Transfer) 是一种软件架构风格，主要用于设计网络应用的接口。

主要原则

  1. 资源标识
  • 每个资源都有唯一的URI标识
  • 使用名词而不是动词表示资源
  • 资源可以是单个或集合，如 /users 和 /users/{id}
  1. HTTP方法语义
  • GET: 获取资源
  • POST: 创建资源
  • PUT: 更新资源（完整更新）
  • PATCH: 更新资源（部分更新）
  • DELETE: 删除资源
  1. 状态码使用
  • 2xx: 成功（200 OK, 201 Created, 204 No Content）
  • 3xx: 重定向
  • 4xx: 客户端错误（400 Bad Request, 404 Not Found）
  • 5xx: 服务器错误
  1. 无状态性
  • 服务器不保存客户端状态
  • 每个请求包含所需的完整信息
  • 有助于提高可伸缩性

最佳实践

  1. 版本控制
/api/v1/users
/api/v2/users
  1. 过滤、排序、分页
/users?page=2&size=10  // 分页
/users?sort=name&order=desc  // 排序
/users?role=admin  // 过滤
  1. 返回格式
json
{
    "code": 200,
    "message": "success",
    "data": {
        "id": 1,
        "name": "张三"
    }
}

4. 浏览器缓存(强缓存/协商缓存) ​

浏览器缓存分为两种：强缓存和协商缓存

强缓存

  • 不需要发送请求到服务端，直接从缓存中读取资源
  • 通过 Expires 和 Cache-Control 控制
  • 返回状态码：200（从缓存中读取）

1. Expires (HTTP/1.0)

客户端:

javascript
fetch('/api/data', {
  headers: {
    'If-None-Match': localStorage.getItem('etag'),
  }
})

服务端:

javascript
app.get('/api/data', (req, res) => {
  res.setHeader('Expires', new Date(Date.now() + 24*60*60*1000).toUTCString());
  res.send(data);
});

2. Cache-Control (HTTP/1.1)

客户端:

javascript
fetch('/api/data', {
  cache: 'force-cache'
})

服务端:

javascript
app.get('/api/data', (req, res) => {
  res.setHeader('Cache-Control', 'max-age=86400');
  res.send(data);
});

协商缓存

  • 需要发送请求到服务端，服务端判断资源是否修改
  • 如果未修改，返回304，使用缓存
  • 如果已修改，返回200和新资源

1. Last-Modified/If-Modified-Since

客户端:

javascript
fetch('/api/data', {
  headers: {
    'If-Modified-Since': localStorage.getItem('lastModified')
  }
})

服务端:

javascript
app.get('/api/data', (req, res) => {
  const lastModified = getFileLastModified();
  const ifModifiedSince = req.headers['if-modified-since'];
  
  if (ifModifiedSince && new Date(ifModifiedSince) >= new Date(lastModified)) {
    res.status(304).end();
  } else {
    res.setHeader('Last-Modified', lastModified);
    res.send(data);
  }
});

2. ETag/If-None-Match

客户端:

javascript
fetch('/api/data', {
  headers: {
    'If-None-Match': localStorage.getItem('etag')
  }
})

服务端:

javascript
app.get('/api/data', (req, res) => {
  const etag = generateETag(data);
  const ifNoneMatch = req.headers['if-none-match'];
  
  if (ifNoneMatch === etag) {
    res.status(304).end();
  } else {
    res.setHeader('ETag', etag);
    res.send(data);
  }
});

5. Cache-Control 的取值 ​

主要的 Cache-Control 指令包括：

  1. 可缓存性：

    • public：响应可以被任何对象缓存

    • private：响应只能被单个用户缓存

    • no-cache：需要进行协商缓存

    • no-store：不缓存任何内容

  2. 过期时间：

    • max-age=<seconds>：设置缓存存储的最大周期
    • s-maxage=<seconds>：覆盖max-age，适用于共享缓存
    • max-stale=<seconds>：客户端愿意接受的最大过期时间

  3. 重新验证：

    • must-revalidate：过期后必须验证
    • proxy-revalidate：要求中间缓存服务器验证

  4. 其他：

    • no-transform：不允许对资源进行转换
    • only-if-cached：只使用缓存内容

6. 常见的 HTTP 状态码以及代表的意义 ​

HTTP状态码分为5类：

  1. 1xx - 信息性状态码：

    • 100 Continue: 继续发送请求
    • 101 Switching Protocols: 切换协议

  2. 2xx - 成功状态码：

    • 200 OK: 请求成功
    • 201 Created: 已创建
    • 204 No Content: 请求成功但无内容返回

  3. 3xx - 重定向状态码：

    • 301 Moved Permanently: 永久重定向
    • 302 Found: 临时重定向
    • 304 Not Modified: 资源未修改

  4. 4xx - 客户端错误状态码：

    • 400 Bad Request: 请求语法错误
    • 401 Unauthorized: 未授权
    • 403 Forbidden: 禁止访问
    • 404 Not Found: 资源不存在
    • 405 Method Not Allowed: 方法不允许

  5. 5xx - 服务器错误状态码：

    • 500 Internal Server Error: 服务器内部错误
    • 502 Bad Gateway: 网关错误
    • 503 Service Unavailable: 服务不可用
    • 504 Gateway Timeout: 网关超时

7. 网络状态 301、302、303 有何区别? ​

这三个状态码都属于重定向，但用途不同：

  1. HTTP状态码301: 永久重定向。表示请求的资源已被永久移动到新的位置，将来任何新的请求都应使用新的URL。大多数浏览器会缓存这个重定向的URL，所以在下次访问旧的URL时，浏览器会直接跳转到新的URL，而不会再向服务器请求。

  2. HTTP状态码302: 临时重定向。表示请求的资源临时移动到新的位置，但未来的请求仍应使用原始的URL。浏览器通常不会缓存这个重定向的URL，所以每次访问旧的URL时，都会向服务器请求，然后服务器再返回新的URL.

  3. HTTP状态码303: 查看其他位置，表示请求的资源存在于另一个URL，应使用GET方法获取。这个状态码主要用于在执行POST、PUT等可能引起服务器状态变化的操作后，将客户端重定向到一个新的资源，避免用户刷新或重复提交表单，

8. 400 和 401、403 状态码 ​

  1. 400 Bad Request：

    • 客户端请求语法错误
    • 服务器无法理解请求
    • 常见原因：参数格式错误、缺少必要参数

  2. 401 Unauthorized：

    • 未经授权的请求
    • 用户需要提供身份认证
    • 通常需要提供用户名和密码

  3. 403 Forbidden：

    • 服务器理解请求但拒绝执行
    • 用户没有访问权限
    • 即使提供认证也不会获得授权

9. Http 和 Https 的区别 ​

主要的区别在于安全性和数据传输方式上，HTTPS比HTTP更加安全，适合用于保护网站用户的隐私和安全，如银行 网站、电子商务网站等。

  • 安全性: HTTP协议传输的数据都是未加密的，也就是明文的，因此使用HTP协议传输的数据可以被任何抓包工具截取并查看。而HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，更为安全。
  • 数据传输方式: HTTP协议的端口号是80，HTTPS协议的端口号是443.
  • 网址导航栏显示: 使用HTTP协议的网站导航栏显示的是"http://"，而使用HTTPS协议的网站导航栏显示的是"https://"。
  • 证书: HTTPS需要到CA申请证书，一般免费证书较少，因而需要一定费用。
  • 网络速度: HTTP协议比HTTPS协议快，因为HTTPS协议需要进行加密和解密的过程。
  • SEO优化: 搜索引擎更倾向于把HTTPS网站排在更前面的位置，因为HTTPS更安全

10. 描述一下 HTTPS 的加密过程 ​

HTTPS加密过程包含对称加密和非对称加密：

  1. 握手阶段：

    • 客户端发送随机数和支持的加密算法
    • 服务器返回随机数和数字证书
    • 客户端验证证书，生成预主密钥
    • 双方生成会话密钥

  2. 数据传输阶段：

    • 使用会话密钥加密通信内容
    • 采用对称加密方式
    • 定期更新会话密钥

  3. 具体步骤：

1. Client Hello: 客户端发送随机数1
2. Server Hello: 服务器发送随机数2和证书
3. 客户端验证证书
4. 客户端生成随机数3(预主密钥)
5. 使用公钥加密随机数3
6. 服务器使用私钥解密得到随机数3
7. 双方使用三个随机数生成会话密钥
8. 开始加密通信

11. Cookie 为了解决什么问题 ​

Cookie 主要解决了 HTTP 无状态的问题：

  1. 会话状态管理：

    • 用户登录状态
    • 购物车
    • 游戏分数

  2. 个性化设置：

    • 用户偏好
    • 主题设置

  3. 浏览器行为跟踪：

    • 分析用户行为
    • 广告推荐

12. Cookie 和 Session 的区别 ​

Cookie (HTTP Cookie)和 Session(会话)都是用于在 Web 应用程序中维护状态和用户身份的两种不同机制:

  • 存储位置:
  • Cookie: Cookie是存储在客户端(通常是用户的浏览器)中的小段文本数据。浏览器会在每次请求中自动发送Cookie到服务器，以便服务器可以识别用户。
  • Session: Session数据通常存储在服务器上，而不是在客户端。服务器为每个用户创建一个唯一的会话，然后在服务器上存储会话数据。
  • 持久性:
  • Cookie: Cookie可以具有持久性，可以设置过期时间。如果没有设置过期时间，Cookie将成为会话Cookie，存在于用户关闭浏览器前的会话期间。
  • Session: 会话数据通常存在于用户活动的会话期间，一旦会话结束(用户退出登录或关闭浏览器)，会话数据通常会被删除。
  • 安全性:
  • Cookie: Cookie数据存储在客户端，可能会被用户篡改或窃取。因此，敏感信息通常不应存储在Cookie中，或者应该进行加密。
  • Session: Session数据存储在服务器上，客户端不可见，因此通常更安全，特别适合存储敏感信息。
  • 服务器负担:
  • Cookie: 服务器不需要维护Cookie的状态，因为它们存储在客户端。每次请求中都包含Cookie，服务器只需要验证Cookie的有效性。
  • Session: 服务器需要维护会话数据，这可能会增加服务器的负担，尤其是在大型应用程序中。
  • 跨多个页面:
  • Cookie: Cookie可以被跨多个页面和不同子域共享，这使得它们适用于用户跟踪和跨多个页面的数据传递。
  • Session: 会话数据通常只在单个会话期间可用，而不容易在不同会话之间共享。
  • 无需登录状态:
  • Cookie: Cookie可以在用户未登录的情况下使用，例如用于购物车或用户首选项。
  • Session: 会话通常与用户的身份验证和登录状态相关，需要用户登录后才能创建和访问会话。

13. TCP(传输控制协议)和 UDP(用户数据报协议)的区别 ​

两种常用的传输层协议，用于在网络中传输数据：

TCP:

  • 一种面向连接的协议，提供可靠的数据传输
  • 通过三次握手建立连接，保证数据的完整性和顺序性
  • 使用流控制、拥塞控制和错误检测等机制来确保数据的可靠传输
  • 适用于需要可靠传输的应用，如文件传输、电子邮件和网页浏览等

UDP:

  • 一种无连接的协议，提供不可靠的数据传输
  • 不需要建立连接，直接将数据包发送给目标地址
  • 没有流控制和拥塞控制机制，也不保证数据的完整性和顺序性
  • 适用于实时性要求较高的应用，如音频、视频和实时游戏等

总结来说，TCP提供可靠的、面向连接的数据传输，适用于对数据完整性和顺序性要求较高的应用；而UDP提供不可靠的、无连接的数据传输，适用于实时性要求较高的应用。选择使用TCP还是UDP取决于应用的需求和特点。

14. TCP 三次握手 ​

三次握手的过程：

  1. 第一次握手：

    • 客户端发送 SYN=1, seq=x
    • 进入 SYN_SENT 状态

  2. 第二次握手：

    • 服务器返回 SYN=1, ACK=x+1, seq=y
    • 进入 SYN_RCVD 状态

  3. 第三次握手：

    • 客户端发送 ACK=y+1, seq=x+1
    • 双方进入 ESTABLISHED 状态
Client                Server
   |                    |
   |-----SYN,seq=x---->|
   |                    |
   |<--SYN,ACK,seq=y---|
   |                    |
   |------ACK--------->|
   |                    |

15. 如果 TCP 变成二次握手会导致的问题 ​

二次握手可能导致以下问题：

  1. 无法确认客户端的接收能力：

    • 服务器无法确认客户端是否能收到响应
    • 可能导致资源浪费

  2. 无法防止历史连接：

    • 旧的连接请求可能被服务器接受
    • 造成连接混乱

  3. 安全性问题：

    • 容易被攻击者利用
    • 可能导致 SYN 洪泛攻击

16. TCP 的四次挥手 ​

四次挥手的过程：

  1. 第一次挥手：

    • 客户端发送 FIN=1, seq=u
    • 进入 FIN_WAIT_1 状态

  2. 第二次挥手：

    • 服务器发送 ACK=u+1, seq=v
    • 进入 CLOSE_WAIT 状态

  3. 第三次挥手：

    • 服务器发送 FIN=1, seq=w
    • 进入 LAST_ACK 状态

  4. 第四次挥手：

    • 客户端发送 ACK=w+1
    • 进入 TIME_WAIT 状态
Client                Server
   |                    |
   |------FIN,seq=u--->|
   |                    |
   |<----ACK=u+1-------|
   |                    |
   |<---FIN,seq=w------|
   |                    |
   |-----ACK=w+1------>|
   |                    |

17. 描述一下 TCP 的拥塞控制 ​

TCP 拥塞控制包含四个算法：

  1. 慢启动：

    • 初始发送窗口设为1
    • 每收到一个ACK，窗口大小翻倍
    • 直到达到慢启动阈值

  2. 拥塞避免：

    • 每个RTT窗口只增加1
    • 线性增长，避免拥塞

  3. 快重传：

    • 收到3个重复ACK
    • 立即重传丢失的报文段

  4. 快恢复：

    • 将慢启动阈值设为当前窗口的一半
    • 窗口大小设为新的阈值
    • 执行拥塞避免算法

18. 什么是跨域？如何解决？ ​

跨域是由浏览器的同源策略造成的，当前域名的 JavaScript 试图访问另一个域名下的资源时会发生跨域问题。

同源的定义

  • 协议相同
  • 域名相同
  • 端口相同

解决方案

  1. CORS（跨域资源共享）
javascript
// 服务器设置响应头
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type
  1. JSONP
javascript
function handleResponse(data) {
    console.log(data);
}
<script src="http://api.example.com/data?callback=handleResponse"></script>
  1. 代理服务器
  • 通过服务器转发请求
  • Nginx反向代理
  1. postMessage
javascript
// 发送消息
window.postMessage('Hello', 'http://receiver.com');
// 接收消息
window.addEventListener('message', function(event) {
    console.log(event.data);
});

19. 同源策略具体限制的内容 ​

同源策略主要限制以下几个方面：

  1. Cookie、LocalStorage 和 IndexDB 的访问：

    • 不同源的页面无法读取这些存储

  2. DOM 访问限制：

    • 不能访问跨域页面的 DOM

  3. AJAX 请求限制：

    • 不能发送跨域 AJAX 请求

例外情况： ​

  1. <script> 标签
  2. <img> 标签
  3. <link> 标签
  4. <video> 和 <audio> 标签

21. XSS 攻击是什么？ ​

XSS（跨站脚本攻击）是一种代码注入攻击。

  1. 存储型XSS

    • 恶意代码存储在数据库中
    • 用户访问页面时执行

  2. 反射型XSS

    • 恶意代码包含在URL中
    • 服务器将恶意代码反射回浏览器

  3. DOM型XSS

    • 不经过服务器
    • 直接在浏览器中执行

防御措施

  1. 输入过滤
javascript
function escapeHTML(str) {
    return str.replace(/[&<>'"]/g, function(tag) {
        const chars = {
            '&': '&amp;',
            '<': '&lt;',
            '>': '&gt;',
            "'": '&#39;',
            '"': '&quot;'
        };
        return chars[tag] || tag;
    });
}
  1. CSP内容安全策略
html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'">
  1. HttpOnly Cookie
javascript
Set-Cookie: name=value; HttpOnly

27. 封装一个 ajax 请求方法 ​

javascript
function ajax(options = {}) {
    // 默认配置
    const defaults = {
        method: 'GET',
        url: '',
        data: null,
        headers: {
            'Content-Type': 'application/json'
        },
        responseType: 'json',
        timeout: 5000,
        withCredentials: false
    };

    // 合并配置
    const config = { ...defaults, ...options };

    // 返回Promise
    return new Promise((resolve, reject) => {
        const xhr = new XMLHttpRequest();

        // 超时处理
        xhr.timeout = config.timeout;
        xhr.ontimeout = () => reject(new Error('Request timeout'));

        // 错误处理
        xhr.onerror = () => reject(new Error('Request failed'));

        // 处理GET请求参数
        if (config.method.toUpperCase() === 'GET' && config.data) {
            config.url += '?' + new URLSearchParams(config.data).toString();
        }

        // 打开连接
        xhr.open(config.method, config.url, true);

        // 设置响应类型
        xhr.responseType = config.responseType;

        // 设置跨域携带凭证
        xhr.withCredentials = config.withCredentials;

        // 设置请求头
        Object.keys(config.headers).forEach(key => {
            xhr.setRequestHeader(key, config.headers[key]);
        });

        // 处理响应
        xhr.onload = function() {
            if (xhr.status >= 200 && xhr.status < 300) {
                resolve(xhr.response);
            } else {
                reject(new Error(`Request failed with status ${xhr.status}`));
            }
        };

        // 发送请求
        xhr.send(config.method === 'GET' ? null : JSON.stringify(config.data));
    });
}

// 使用示例
ajax({
    url: '/api/data',
    method: 'POST',
    data: { name: 'test' }
}).then(response => {
    console.log(response);
}).catch(error => {
    console.error(error);
});

29. fetch 与 XMLHttpRequest 的区别 ​

主要区别：

  1. Promise 基础：

    • Fetch 基于 Promise
    • XHR 基于回调

  2. 请求中断：

    • Fetch 需要 AbortController
    • XHR 可以直接调用 abort()

  3. 请求进度：

    • Fetch 不能监测上传进度
    • XHR 可以监测上传和下载进度

  4. 错误处理：

    • Fetch 只有网络错误才会 reject
    • XHR 状态码错误就会触发 onerror

  5. 默认行为：

    • Fetch 默认不发送 Cookie
    • XHR 默认发送 Cookie

34. Server-Sent Events (SSE) ​

SSE 是一种服务器推送技术，允许服务器向客户端推送数据。

  1. 单向通信（服务器到客户端）
  2. 基于HTTP协议
  3. 自动重连机制
  4. 事件ID和断点续传
  5. 纯文本数据传输

应用场景

  1. 实时新闻推送
  2. 股票价格更新
  3. 社交媒体feed流
  4. 实时通知系统

35. Server-Sent Events 示例代码 ​

客户端代码

javascript
const evtSource = new EventSource('/events');

// 连接建立时
evtSource.onopen = () => {
    console.log('Connection opened');
};

// 接收消息
evtSource.onmessage = (event) => {
    console.log('Received:', event.data);
};

// 自定义事件
evtSource.addEventListener('update', (event) => {
    console.log('Update:', event.data);
});

// 错误处理

    console.error('Error:', error);
    evtSource.close();
};

服务器代码(Node.js)

javascript
const express = require('express');
const app = express();

app.get('/events', (req, res) => {
    // 设置SSE所需的响应头
    res.writeHead(200, {
        'Content-Type': 'text/event-stream',
        'Cache-Control': 'no-cache',
        'Connection': 'keep-alive'
    });

    // 发送消息
    const sendMessage = () => {
        res.write(`data: ${JSON.stringify({time: new Date()})}\n\n`);
    };

    // 发送自定义事件
    const sendUpdate = () => {
        res.write(`event: update\ndata: {"status": "updated"}\n\n`);
    };

    // 定时发送消息
    const timer = setInterval(sendMessage, 1000);
    const updateTimer = setInterval(sendUpdate, 5000);

    // 连接关闭时清理
    req.on('close', () => {
        clearInterval(timer);
        clearInterval(updateTimer);
    });
});

36. SSE与WebSocket的区别 ​

1. 通信方式：

  • SSE：单向通信（服务器到客户端）
  • WebSocket：双向通信

2. 协议：

  • SSE：基于HTTP协议
  • WebSocket：独立的WS协议

3. 断线重连：

  • SSE：自动重连机制
  • WebSocket：需要手动实现重连

4. 数据格式：

  • SSE：仅支持文本数据
  • WebSocket：支持文本和二进制数据

5. 使用场景：

  • SSE：适合服务器推送数据的场景
  • WebSocket：适合需要双向通信的场景

6. 浏览器支持：

  • SSE：主流浏览器都支持
  • WebSocket：同样广泛支持，但可能需要降级方案

37. HTTP/2.0 特性 ​

1. 多路复用：

  • 在一个TCP连接上可以同时传输多个HTTP请求和响应
  • 解决了HTTP/1.1的队头阻塞问题
  • 减少了TCP连接数

2. 二进制分帧：

  • 将消息分割为更小的帧
  • 支持请求和响应的并行传输
  • 提高了传输效率

3. 服务器推送：

http
Link: </styles.css>; rel=preload; as=style
  • 服务器可以主动推送资源
  • 减少客户端请求次数
  • 提高页面加载速度

4. 头部压缩：

  • 使用HPACK算法压缩头部
  • 减少了重复头部的传输
  • 降低了带宽占用

5. 优先级和依赖性：

  • 可以设置请求的优先级
  • 优化关键资源的加载
  • 提高用户体验

示例配置(Nginx)：

nginx
server {
    listen 443 ssl http2;
    server_name example.com;
    
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    
    # HTTP/2 specific settings
    http2_push_preload on;
    http2_max_concurrent_streams 128;
}

性能对比：

  1. 延迟降低：

    • HTTP/1.1：多个TCP连接
    • HTTP/2：单个TCP连接复用

  2. 资源加载：

    • HTTP/1.1：6-8个并发连接
    • HTTP/2：无限制（实际受服务器限制）

  3. 头部开销：

    • HTTP/1.1：每次请求都发送完整头部
    • HTTP/2：仅发送差异头部
