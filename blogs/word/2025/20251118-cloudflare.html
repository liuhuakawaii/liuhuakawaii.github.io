一、先给结论（方便快速决定）

如果你已经把前端部署在 Cloudflare Pages 且仅需一个简单代理 /api/v2 来隐藏 token：优先选 Pages Functions（最简单、同仓部署、同域、易调试）。

如果你需要更复杂的路由、多个独立服务、或想把代理独立管理：选 Cloudflare Worker（独立 Worker）（更灵活、可复用、适合大型场景）。

下面先把概念讲清，再分别给出两套完整可运行示例（Pages Functions 版本与独立 Worker 版本），包含本地调试、如何写 secret、如何部署、如何在前端调用、注意事项与常见问题排查。

二、概念与差异速览（图文并茂口语化）

Cloudflare Worker：一个独立的边缘 JavaScript 环境（类似在全球有很多小型 serverless 节点），你可以写任意后端逻辑、绑定任意域名或 route。通常通过 wrangler 管理与部署。 优点：独立、灵活、强；缺点：对初学者多一步配置（wrangler、route、zone_id 等）。

Pages Functions：Cloudflare Pages 内置的“函数目录”方案，意味着你的静态站（Pages）与后端函数放在同一个代码仓库 & 一次部署。文件放在 functions/（或 Pages 要求的路径），Pages 会自动部署并将路径映射到函数。 优点：极简体验（同一 repo、一键部署），对前端工程师友好；缺点：在多项目/微服务场景下不够独立。

三、环境准备（通用）

Node.js（建议 >=18）

Cloudflare 帐号（已注册）

你已有前端项目（React + Vite），并且已部署/准备部署到 Cloudflare Pages（可选）

本地终端、编辑器（VSCode 等）

安装/登录 wrangler（用于独立 Worker 或 pages secret 操作）：

bash
npm install -g wrangler@latest   # 可选：也可用 npx 执行
wrangler login

四、重要概念：Variable vs Secret（必须记住）

Variable（env / vars）：可以是明文，用于非敏感配置（构建时注入、或 env 运行时使用）。不要放敏感 token。

Secret（加密）：加密存储，只能在 Worker/Pages Function 的运行时读取，不能泄露在构建或浏览器端。把 Bearer token 放到 Secret 里。

在 CLI：

Pages secret（绑定到 Pages 项目）：

bash
npx wrangler pages secret put API_TOKEN --project-name <your-pages-project>

Worker secret（独立 Worker）：

bash
echo "MY_TOKEN" | npx wrangler secret put API_TOKEN

五、方案 A：Pages Functions（推荐你的场景）

场景适用：你的网站已经部署在 Cloudflare Pages，想在同域下把 /api/v2/* 代理到后端并隐藏 Bearer token。页面与函数放同仓，一起部署最方便。

  1. 文件位置 & 约定（以常见 Pages 项目结构为例）

在你的项目里（与 dist / build 输出并列）创建：

bash
project-root/
  ├─ src/            # React 源码（不受影响）
  ├─ functions/
  │   └─ api/
  │       └─ v2/
  │           └─ [...path].js   # catch-all，处理 /api/v2/*
  ├─ package.json
  └─ other files...

functions/api/v2/[...path].js（完整代码）：

javascript
// functions/api/v2/[...path].js
export async function onRequest(context) {
  // context 包含 request, env, params, etc.
  const { request, env, params } = context;
  const url = new URL(request.url);
  // params.path 是 catch-all 捕获到的部分（数组形式或字符串，依据 runtime）
  // 这里我们直接利用 pathname 替换更简单：
  const upstreamBase = env.GLOBAL_UPSTREAM_BASE || 'https://api.example.com';
  if (!url.pathname.startsWith('/api/v2')) {
    return new Response('Not Found', { status: 404 });
  }

  // 映射路径： /api/v2/foo/bar -> /v2/foo/bar
  const upstreamPath = url.pathname.replace(/^\/api\/v2/, '/v2');
  const upstreamUrl = upstreamBase + upstreamPath + url.search;

  const apiToken = env.API_TOKEN; // Pages secret / environment binding
  if (!apiToken) {
    return new Response('Server misconfigured: API_TOKEN missing', { status: 500 });
  }

  // 复制原请求的 headers 和 body
  const headers = new Headers(request.headers);
  headers.set('Authorization', `Bearer ${apiToken}`);
  headers.delete('Host');

  const init = {
    method: request.method,
    headers,
    body: (request.method === 'GET' || request.method === 'HEAD') ? undefined : request.body,
    redirect: 'manual'
  };

  const resp = await fetch(upstreamUrl, init);

  // 处理 CORS（如果同域部署，严格限制 origin 更安全）
  const respHeaders = new Headers(resp.headers);
  respHeaders.set('Access-Control-Allow-Origin', '*');
  respHeaders.set('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE,OPTIONS');
  respHeaders.set('Access-Control-Allow-Headers', 'Content-Type,Authorization');

  return new Response(resp.body, {
    status: resp.status,
    statusText: resp.statusText,
    headers: respHeaders
  });
}

说明要点

env.API_TOKEN：在 Pages 项目里通过 CLI 或 Dashboard 把 secret 加入（见下）。

Path catch-all：Pages Functions 支持以 [...] 方式捕获任意子路径。

CORS：示例使用 *，上线请把 * 换成你的域名以更安全（例如 https://mulerun-rodin.pages.dev）。

场景适用：你想把代理做成独立服务、需要自定义路由或多个站点共用同一个代理，或想独立运维。

  1. 项目结构（最小）
bash
my-wrangler/
  ├─ src/
  │   └─ index.js
  ├─ wrangler.toml
  └─ package.json

src/index.js（模块化 Worker）：

js
export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    if (!url.pathname.startsWith('/api/v2')) {
      return fetch(request); // passthrough 或 404
    }

    const upstreamBase = env.GLOBAL_UPSTREAM_BASE || 'https://api.example.com';
    const upstreamPath = url.pathname.replace(/^\/api\/v2/, '/v2');
    const upstreamUrl = upstreamBase + upstreamPath + url.search;

    const apiToken = env.API_TOKEN;
    if (!apiToken) {
      return new Response('Server misconfigured: API_TOKEN missing', { status: 500 });
    }

    const headers = new Headers(request.headers);
    headers.set('Authorization', `Bearer ${apiToken}`);
    headers.delete('Host');

    const init = {
      method: request.method,
      headers,
      body: (request.method === 'GET' || request.method === 'HEAD') ? undefined : request.body,
      redirect: 'manual'
    };

    const upstreamResp = await fetch(upstreamUrl, init);

    const respHeaders = new Headers(upstreamResp.headers);
    respHeaders.set('Access-Control-Allow-Origin', '*');
    respHeaders.set('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE,OPTIONS');
    respHeaders.set('Access-Control-Allow-Headers', 'Content-Type,Authorization');

    return new Response(upstreamResp.body, {
      status: upstreamResp.status,
      statusText: upstreamResp.statusText,
      headers: respHeaders
    });
  }
};

七、本地开发细节：.dev.vars 与 Vite 本地代理

开发时你不该把真实 token 放到仓库。两种常用方法：

  1. 用 .dev.vars（仅本地、NOT committed）来提供 env：
bash
API_TOKEN = "LOCAL_FAKE_TOKEN"
GLOBAL_UPSTREAM_BASE = "https://api.example.com"

wrangler dev 会读取 .dev.vars 并注入 env。 2. Vite 本地 dev 时，用 server.proxy 把 /api/v2 转发到 wrangler dev 本地地址（http://127.0.0.1:8787），从而在本地体验同域调用：

js
// vite.config.js
export default {
  server: {
    proxy: {
      '/api/v2': 'http://127.0.0.1:8787'
    }
  }
}

八、前端（React）如何调用代理

前端只需要像调用本地 API 一样调用相对路径，不带任何敏感 token：

js
// src/api.js
export async function callApi(path, data) {
  const res = await fetch(`/api/v2/${path}`, {
    method: data ? 'POST' : 'GET',
    headers: { 'Content-Type': 'application/json' },
    body: data ? JSON.stringify(data) : undefined,
    credentials: 'same-origin',
  });
  if (!res.ok) { throw new Error(`HTTP ${res.status}`); }
  return res.json();
}

注意在本地开发时，若你没有运行 wrangler dev，可以在 Vite 中配置代理把 /api/v2 指向真实上游（仅用于开发，切勿把真实 token 放在前端 env 中）。

九、CORS、安全与注意事项

CORS：如果你把前端与 Functions/Worker 放在同域（pages domain），通常无需 CORS；如果跨域访问，需要在 Worker/Function 返回 Access-Control-Allow-Origin 等头。生产环境请替换 * 为具体域名。

不要把 token 写成 VITE_ 前缀的 env：Vite 中以 VITE_ 开头的环境变量会被打包进前端代码并能被浏览器看到 —— 绝对不要放敏感信息。

不要把 secret 写到 wrangler.toml 的 vars：那是明文。应该使用 wrangler secret put 或 Pages 的 secret 功能。

鉴权：Worker/Function 能隐藏后端 token，但仍需做业务级鉴权（比如用户身份验证、访问频率控制等），否则任何人都能调用代理（可以限制 origin / referer，但这些头可伪造）。更安全的方式是让前端先登录并带上用户 token（短期 token）进行二次校验，Worker 再用服务器 token 和上游服务沟通。 十、完整操作清单（Copy & Paste）

十一、排错清单（常见问题）

请求返回 500，提示 API_TOKEN missing：说明 env 没注入 secret，确认你在 Pages Dashboard 或用 wrangler pages secret put 提交了 API_TOKEN。

本地 wrangler dev 看不到 secret：wrangler dev 不会自动读取 Dashboard 的 secret；用 .dev.vars 本地模拟或用 wrangler secret put（这会上传到 Cloudflare）。

CORS 报错（浏览器）：检查 Worker/Function 是否在 OPTIONS 请求中返回 Access-Control-Allow-* 头。对 preflight（OPTIONS）要返回 200 并包含允许的 headers。

404：检查 functions 文件路径与请求路径是否匹配（Pages Functions 路径映射严格，catch-all 文件名要写对）。
